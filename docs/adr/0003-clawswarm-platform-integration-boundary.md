# ADR 0003：ClawSwarm 以独立协作服务接入平台，而不是直接并入主应用源码

## 状态

已采纳

## 背景

平台已经从单机运维后台演进为面向最终用户的多租户 OpenClaw 平台。

新的目标不是简单增加一个聊天窗口，而是让每个注册用户都能拥有自己的龙虾，并进一步使用多 Agent 协作能力。

`ClawSwarm` 提供了现成的群智协作能力，但它本身是一个完整产品，包含：

- Vue 前端 `web-client`
- Python 后端 `scheduler-server`
- OpenClaw `channel` 插件
- 原生登录、消息、任务、实例、设置等功能模块

同时，上游许可证为 `GPL-3.0`，当前主仓库为 `MIT`。

## 决策

采用“**独立 ClawSwarm 服务 + 平台适配层 + 平台原生入口**”的集成方式，而不是：

- 直接把 `ClawSwarm` 原生前端整块嵌入为最终方案
- 直接把 `ClawSwarm` 源码大面积并入 `apps/admin-ui`

具体落地原则：

1. `ClawSwarm` 作为独立协作运行时服务存在。
2. `http://localhost:18889` 保持为最终用户唯一入口。
3. 平台用户体系是唯一主身份源。
4. 平台龙虾与平台模型是主数据源。
5. 平台通过适配层完成租户过滤、身份映射、API 代理与 UI 整形。
6. 对尚未平台原生覆盖的功能，可临时保留受控兼容视图，但不把它作为最终架构。
7. 对上游服务如需修改，采用受控 fork，并保留 GPL 合规边界。

## 备选方案

### 方案 1：直接嵌入上游原生前端

未采纳。

原因：
- 登录体系冲突
- 多租户隔离不稳
- UI 风格割裂
- 最终仍需二次平台化

### 方案 2：直接合并上游源码到主仓

未采纳。

原因：
- 上游升级困难
- 维护成本高
- 许可证边界复杂
- 与当前仓库节奏不匹配

### 方案 3：独立服务 + 平台适配层 + 平台原生入口

采纳。

原因：
- 上游能力可复用
- 平台主数据与租户模型可控
- GPL 边界更清晰
- 可以按阶段推进，不必一次性推翻现有平台

## 后果

### 正面结果

- 平台可以完整承接群智协作能力。
- 用户仍然只使用你的平台，不需要理解第二套系统。
- 未来可以分阶段把消息、任务、实例、设置逐步重构成统一体验。
- 上游 ClawSwarm 升级仍然保留可操作空间。

### 代价

- 需要新增适配层与映射表。
- 需要维护平台与上游服务之间的同步逻辑。
- 短期内会同时存在“平台原生视图”和“兼容视图”两类页面。
- 需要补充 GPL 相关合规说明与发布策略。

## 备注

这个 ADR 与以下目标强相关：

- 多租户平台化
- 用户工作台与管理员后台分离
- Docker 部署友好
- 后续向“每个用户一套可协作龙虾系统”演进
